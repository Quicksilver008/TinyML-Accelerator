Architecture Info

        
      

      

        

          
Files

          
integration/pcpi_demo/rtl/pcpi_tinyml_accel.v is the wrapper. accel_standalone/rtl/matrix_accel_4x4_q5_10.v is the accelerator top. accel_standalone/rtl/pe_cell_q5_10.v defines the PE multiply-accumulate behavior.

        

        

          
System Contract

          
The CPU presents one custom instruction. The wrapper decodes it, latches pcpi_rs1 -> base_a and pcpi_rs2 -> base_b, reads 16 A words and 16 B words, starts the accelerator, waits for accel_done, writes 16 C words to 0x0000_0200, then returns c00 on pcpi_rd.

        

        

          
Instruction Encoding

          
Wrapper matches a custom instruction with opcode = 7'b0001011 (custom-0), funct3 = 3'b000, and funct7 = 7'b0101010.

        

        

          
Fixed-Point Format

          
Inputs and outputs use Q5.10 fixed point in signed 16-bit values. In each PE, product_full = x_in * y_in and product_q5_10 = product_full >>> 10 before accumulation into z_acc.

        

        

          
Timing of the Accelerator Transaction

          
Wrapper FSM runs S_IDLE -> S_LOAD_A -> S_LOAD_B -> S_KICK -> S_WAIT_ACC -> S_STORE_C -> S_RESP. The accelerator itself is configured for COMPUTE_CYCLES = 10 for the 4x4 systolic schedule.

        

        

          
Domains

          
    
            
  • CPU domain: originates pcpi_valid, pcpi_insn, pcpi_rs1, pcpi_rs2.
    • 
            
  • Wrapper domain: decodes, stalls CPU, reads A/B from memory, kicks accelerator, writes C, returns response.
    • 
            
  • Accelerator domain: latches operands, issues wavefront, computes c_flat, raises done.
    • 
            
  • Memory domain: serves wrapper requests through accel_mem_valid/we/addr/wdata/rdata/ready.
    • 
          

        

        

          
CPU Stall and Handoff

          
CPU side sees pcpi_wait = 1 through load, kick, wait, and store phases. It is released only when the wrapper raises pcpi_ready / pcpi_wr in S_RESP. After that, normal instruction flow resumes. This is the handoff point between accelerator-assisted execution and regular CPU execution.

        

        

          
Key Signal Groups

          
    
            
  • PCPI request: pcpi_valid, pcpi_insn, pcpi_rs1, pcpi_rs2
    • 
            
  • PCPI response: pcpi_wait, pcpi_ready, pcpi_wr, pcpi_rd
    • 
            
  • Memory path: accel_mem_valid, accel_mem_we, accel_mem_addr, accel_mem_wdata, accel_mem_rdata, accel_mem_ready
    • 
            
  • Accelerator path: accel_start, accel_busy, accel_done, c_flat, accel_cycle_count
    • 
          

        

      

    

  


  

    

    

      

        
CPU Core Info

        
      

      

        

          
PicoRV32 in This Project

          
PicoRV32 is not a classic 5-stage pipeline core. It is an FSM-based RV32 core that exposes the PCPI interface. In this project it provides the custom instruction request and is stalled externally by the wrapper through pcpi_wait.

How the CPU Stalls Here

          
    
            
  • The CPU presents pcpi_valid, pcpi_insn, pcpi_rs1, and pcpi_rs2.
    • 
            
  • If the wrapper matches the custom instruction, it raises pcpi_wait.
    • 
            
  • While pcpi_wait is high, the CPU does not retire this instruction and does not move on to the next one.
    • 
            
  • Once wrapper finishes read, compute, and writeback, it raises pcpi_ready and pcpi_wr.
    • 
            
  • The CPU then consumes pcpi_rd, retires the instruction, and resumes normal flow.
    • 
          

        

        

          
Handoff Behavior

          
Handoff means returning from accelerator-assisted execution back to ordinary CPU execution cleanly. In this project that handoff happens in S_RESP after writeback is complete and the wrapper produces the PCPI response.
